# 智能招聘 Agent — UI 原型实现说明文档

> 版本: v1.0 | 日期: 2026-07-07 | 原型文件目录: `ui/`
> 本文档基于 UI 原型 HTML 文件编写，供前后端开发团队对照实现。所有页面均为单文件 HTML（内含 CSS + JS），可直接浏览器打开预览。

## 一、项目概述

### 1.1 产品定位
HR 面向的智能招聘 Agent 系统，通过自然语言对话驱动招聘全流程。

### 1.2 技术架构规划
- **前端**: React 18 + Ant Design 5 + TypeScript
- **后端**: FastAPI + LangGraph
- **数据库**: PostgreSQL 15 + Redis 7
- **Agent 推理框架**: R-P-R-A-R 思维链（Reason → Reflect → Plan → Reflect → Act → Reflect）

### 1.3 文件结构
```
ui/
├── page-1-chat-task-center.html     # 对话式任务中心（核心入口）
├── page-1b-task-detail.html         # 任务详情（岗位招聘全过程）
```

### 2.6 通用交互模式
- **Toast 通知**: 所有操作反馈通过 Toast 组件（顶部居中弹出，3 秒自动消失），支持 success / info / warning / error 四种类型
- **Modal 弹窗**: 居中弹出，带半透明遮罩，支持 ESC 键和点击遮罩关闭。同一时间只允许一个弹窗
- **Tab 切换**: 滑动指示器动画，内容区 fade 切换
- **表单验证**: 必填字段标红色星号，空值提交时 toast 提示

## 三、R-P-R-A-R 流式对话架构

### 3.1 架构说明
Agent 推理链路：Reason → Reflect → Plan → Reflect-Plan → Act → Reflect-Act

### 3.2 消息类型定义

前端需支持以下消息类型的渲染：

| 类型 | CSS 类名 | 视觉样式 | 说明 |
|------|---------|---------|------|
| Reason（推理） | `.msg-reason` | 深灰背景 + 左侧灰色边框 | Agent 分析用户输入，提取关键信息 |
| Reflect（反思） | `.msg-reflect` | 浅蓝背景 + 左侧蓝色边框 | 自我审视推理结果 |
| Plan（计划） | `.msg-card-plan` | 白色卡片 + 步骤列表 + 3 个操作按钮 | 结构化执行步骤，含预估时间 |
| Tool Call（工具调用） | `.msg-tool-call` | 可折叠卡片，running/done 两种状态 | 显示工具名称、调用原因、参数 |
| Progress（进度） | `.msg-progress` | 进度条 + 实时计数（如 "15/30"） | 渐变填充动画 |
| Result（结果） | `.msg-card-result` | 结果卡片 + 操作栏（查看详情/推送/导出） | 最终执行结果 |
| Reflect-Act（执行反思） | `.msg-reflect` | 同 Reflect 样式 | 总结执行效果 |
| Warning（警告） | `.msg-card-warning` | 橙色背景警告条 | 解析失败等异常 |
| System（系统消息） | `.system-message` | 灰色居中小字 | 系统提示（岗位切换等） |

### 3.3 Plan 卡片交互
Plan 卡片包含 3 个操作按钮：
- **确认执行**: 禁用所有按钮 → 按计划顺序执行 → 显示执行过程（工具调用 + 进度条 + 结果）
- **调整条件**: 展开条件输入框 → 用户修改 → 确认后重新生成计划
- **跳过解析直接评分**: 跳过部分步骤 → 直接执行评分引擎

### 3.4 后端 SSE 事件映射
后端通过 Server-Sent Events 推送以下事件类型，前端需逐个渲染：

```json
// 前端接收格式（参考）
{ "type": "reason", "content": "分析用户指令..." }
{ "type": "reflect", "content": "已确认岗位..." }
{ "type": "plan", "steps": [...], "estimated_time": "43秒" }
{ "type": "tool_call", "tool": "ResumeParser", "reason": "...", "params": "..." }
{ "type": "progress", "current": 15, "total": 30, "label": "正在处理简历" }
{ "type": "result", "content": "已完成筛选..." }
```

## 四、页面详细说明 — page-1 对话式任务中心

**文件**: `ui/page-1-chat-task-center.html`
**定位**: 产品核心入口，统一对话界面

### 4.1 页面结构
- **顶部岗位选择器**: `<select>` 下拉框，选项包括 5 个岗位（后端开发工程师-简历筛选中、高级产品经理-JD编写中、数据分析师-需求解析中、UI设计师-待处理、前端开发工程师-已归档）。切换时更新标题/阶段 badge 并插入系统提示消息。
- **Chat Stream 聊天消息流**: 预置 12 条历史消息，支持所有消息类型渲染。
- **底部输入区**: textarea（auto-resize）+ 发送按钮 + 快捷操作芯片（生成JD/筛选简历/查看报告/任务切换，点击填入意图文本到输入框，不跳转页面）。

### 4.2 交互功能清单

| # | 元素 | 交互行为 | 前端实现 |
|---|------|---------|---------|
| 1 | 岗位选择下拉框 | 切换岗位 → 更新标题/状态/阶段 badge → 追加系统消息 | change 事件监听 |
| 2 | 发送按钮 / Enter | 清空输入框 → 追加用户气泡 → 触发 simulateAgentReply | click + keydown 事件 |
| 3 | textarea | 输入时自适应高度 | input 事件 + scrollHeight 计算 |
| 4 | Plan 确认执行 | 禁用按钮 → 依次显示工具调用/进度/结果 → 完成 | 链式 setTimeout 模拟 |
| 5 | Plan 调整条件 | 展开输入框 → 确认/取消 | 切换 CSS display |
| 6 | Plan 跳过解析 | 跳过步骤直接执行评分 | 禁用按钮 + 直接显示结果 |
| 7 | 快捷芯片 | 填入意图文本到输入框并聚焦 | click → input.value = intent |
| 8 | 顶部菜单 | 显示/隐藏下拉菜单 | click toggle |
| 9 | 工具调用卡片 header | 展开/折叠详情 | click toggle expanded |

### 4.3 模拟数据（3 种场景）

**场景 A — 简历筛选**（关键词：简历、筛选）
- Reason: 分析用户指令：需要对接收的候选人简历进行筛选评分
- Reflect: 已确认岗位 JD 包含5个核心维度要求，候选简历共30份
- Plan: 4 步（解析器/评分引擎/生成报告/推送）
- Tool Calls: ResumeParser（30份简历）、ScoringEngine（5维度评分）
- Progress: 30 份简历处理进度
- Result: 推荐8位，待定12位，淘汰10位

**场景 B — JD 生成**（关键词：JD、职位、岗位、生成）
- Reason: 需要生成一份结构化JD文档
- Plan: 4 步（检索模板/定制要求/生成文档/合规校验）
- Tool Calls: TemplateMatcher、JDComposer、ComplianceChecker

**场景 C — 通用**（其他）
- Reason: 通用需求分析
- Plan: 4 步（理解意图/检索数据/生成方案/输出结果）
- Tool Calls: IntentParser

## 十五、开发注意事项

### 15.1 前端开发
- 所有交互已通过 mock 数据实现，可直接对照 HTML/CSS/JS 实现为 React 组件
- 组件拆分建议：Sidebar（全局复用）、Toast、Modal、TagInput、ProgressBar、PlanCard、ToolCallCard、CandidateCard 等
- R-P-R-A-R 消息流需对接后端 SSE，消息类型使用 `type` 字段区分
- 表格分页、筛选、排序逻辑可直接参考原型 JS 实现
- 假数据结构已在本文档中定义，可转为 TypeScript 接口

### 15.2 后端开发
- LLM 模型配置需实现 OpenAI 兼容 API 适配层（支持 GPT-4o / Claude / DeepSeek / 国产模型）
- Agent 使用 LangGraph 实现 R-P-R-A-R 推理链，通过 SSE 推送每种消息类型
- 简历解析支持 PDF/Word 格式，返回结构化 JSON（基本信息 + 教育背景 + 工作经历 + 技能 + 项目经验）
- 评分引擎需支持可配置的多维度权重，总分 100 分制
- 推送模块需对接企业 IM（飞书/钉钉/企业微信）

### 15.3 数据库
- 核心实体：Task（任务）、JD（职位描述）、Resume（简历）、Candidate（候选人）、Score（评分）、Push（推送）、Feedback（反馈）
- JD ↔ Candidate 双向关联，通过 Score 记录链接
- 简历数据支持多条工作经历、教育背景、项目经验（一对多关系）
